# Code Review: tiny-Engine

> 基于 2026-05-29 全量代码分析

---

## 高优先级

### 1. 🚨 `detach()` 的线程不安全（`core/src/raft/node.cpp:76-115`）

每次出站 RPC 都 `new std::thread(...).detach()`。三个问题：
- **无法安全关闭** — 服务端析构后，仍在运行的 detached 线程会访问已销毁的 `stubs_`、`mutex_`、`raft_`
- **无法取消** — 如果某个 follower 已挂，RPC 会阻塞到 500ms 超时，而 Leader 无法跳过它
- **线程爆炸** — 每个心跳周期对每个 follower 都 spawn 一个线程（3 节点每 20ms 可能 spawn 2 个线程）

**建议**：改用 `grpc::CompletionQueue` + 固定线程池（或 `asio::io_context`），让 RPC 调用可取消且生命周期可控。

### 2. 🚨 `LoadSnapshot` 是空操作（`core/src/server/server.cpp:106-125`）

C++ 端的 `LoadSnapshot` 直接返回空 `done` chunk，没有读取磁盘上的 snapshot 返回给 Java。这意味着**崩溃恢复完全不可用**。

**建议**：实现 chunked read from `FileStorage::LoadSnapshot`，拆分成 `SnapshotChunk` 流式返回。

### 3. 🚨 Java 端 committed stream 没有重连逻辑（`coordinator/src/main/java/engine/coordinator/StateMachineDriver.java:78-83`）

```java
subscription = raftClient.subscribeCommitted(startIdx)
    .subscribe().with(
        this::applyEntry,
        err -> LOG.errorf(err, "Committed stream error; will attempt reconnect")
    );
```

错误回调只打了日志，没有重连。如果 C++ 节点重启，Java 端永久停止处理 committed 条目。

**建议**：使用 Mutiny 的 `onFailure().retry()` 或手动实现指数退避重连。

### 4. 🚨 `ApplyCommand` 丢失了 `term` 信息（`core/src/raft/raft.h:25-28`，`core/src/main.cpp:67`）

```cpp
struct ApplyCommand {
    std::vector<std::byte> data;
    unsigned long long     index = 0;
};
// main.cpp: 传了 0
raft_log_ptr->OnCommitted(m.index, 0, m.data);
```

`ApplyCommand` 没有 term 字段，`main.cpp` 硬编码传 `0`。这使得 Java 端永远拿不到条目的正确 term。

**建议**：给 `ApplyCommand` 加上 `term` 字段。

## 中优先级

### 5. gRPC 线程模型问题 — `RaftLogClient.propose()` 阻塞调用

`coordinator/src/main/java/engine/client/RaftLogClient.java:35-38` 使用 `blockingStub`，如果被 event-loop 线程调用会阻塞 Quarkus 的 event loop。

**建议**：改用 `asyncStub` + `CompletableFuture`，与 `StateMachineDriver` 其他部分风格一致。

### 6. `StateMachineDriver.subscribeCommitted` 没有背压

流式回调直接在 main apply 线程上执行所有操作（反序列化 protobuf、写 KVStore、触发 snapshot）。如果 replay 大量历史日志，可能 OOM 或阻塞 Mutiny 的事件线程。

**建议**：使用 `emitOn` 切换到专用线程池，或限制缓冲窗口。

### 7. RaftNode 析构顺序有隐患（`core/src/raft/node.cpp:33`，`core/src/server/server.cpp:137-157`）

```cpp
RaftNode::~RaftNode() { ticker_.request_stop(); }
```

只停了 ticker 线程，但 `GrpcServer::Shutdown()` 在 main 的 signal handler 中调用。如果 gRPC server 在 `RaftNode` 之后析构，in-flight 的 RPC handler（`RequestVote`、`AppendEntries` 等）可能访问已释放的 `RaftNode`。

**建议**：`GrpcServer` 应先 `Shutdown` 再析构 `RaftNode`；`RaftNode` 析构时应确保所有 in-flight RPC 已完成。

### 8. `LeaseManager.checkExpiry()` 在所有节点上运行

`@Scheduled(every = "1s")` 在三个 C++ 节点对应的三个 coordinator 实例上都跑。非 leader 节点也会尝试 propose lease revocation，虽然会被 Raft 拒绝，但浪费资源且产生无用日志。

**建议**：检查 `status().isLeader`，只在 leader 上执行过期检查。

### 9. `WatchManager` 的 watcher 会泄漏

`ConcurrentHashMap<Long, WatchEntry>` 没有清理机制。如果 client 注册 watch 后断开连接但不发送 `CancelRequest`，entry 永远留在 map 中。

**建议**：给 watch 加 TTL 或使用 gRPC Context 监听 client 断开事件。

### 10. `RaftLogServiceImpl` 的 committed queue 没有上限

`std::deque<CommittedEntry>` 无限增长。如果 Java 消费慢或断连，C++ 内存会持续增长。

**建议**：加一个 cap，超出时丢弃旧条目或阻塞 producer。

## 低优先级

### 11. 测试覆盖率不足

现有 5 个单测只覆盖了 Raft 状态机的基本路径。缺少：
- 网络分区/少数派隔离场景
- 日志冲突消解
- Snapshot install
- gRPC 集成测试
- Java 端单元测试（`CoordinatorTest.java` 是空壳）

### 12. 没有 TLS

gRPC 全部使用 `InsecureServerCredentials` / `InsecureChannelCredentials`。

### 13. C++ 构建中包含了 coordinator.proto

`CMakeLists.txt` 对三个 proto 都生成代码，但 `coordinator.proto` 只在 Java 端使用。增加不必要的编译时间。

### 14. 配置无法外部化

超时参数（`kTickInterval`、`kHeartbeatInterval`、`kElectionTimeoutMin/Max`）硬编码在 `core/src/raft/config.h` 中，无法运行时调整。
